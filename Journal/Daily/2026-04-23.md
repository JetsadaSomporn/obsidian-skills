### Session 2: SLA Alert Fix

- เริ่มจากการอ่าน `SLAMonitorWorker.cs` เพื่อทำความเข้าใจ Logic การแจ้งเตือน SLA ปัจจุบัน
- พบว่า Logic หลัก: SLA >= 80% → ยิง `SLA-WARN` (Medium), SLA > 100% → ยิง `SLA-OVER` (High), สแกนทุก 5 นาที
- **พบ merge conflict 4 จุด** ใน `SLAMonitorWorker.cs` + config `Enabled = false` default + ไม่มี appsettings section
- วิเคราะห์ root cause ของ Noisy Alert:
  1. `AlertRepository` dedup ใช้ exact Action match (`=`) แต่ Action เปลี่ยนทุกรอบ (minutes เปลี่ยน)
  2. `AlertRepository` dedup เช็คเฉพาะ `StatusTo = 'Unread'` — user อ่านแล้วจะยิงใหม่
  3. PM query ไม่มี age filter → สแกน PM เก่าๆ ทุกรอบ
  4. `spAlertInsert` ไม่มี dedup ภายใน → ทุกครั้ง = INSERT ใหม่เสมอ
- วิเคราะห์ flow ข้อมูล: `spAlertInsert` → `AlertTemp_QBuffer` → `AlertQ` → `AlertTemp_ToBuffer` → `AlertTo` → `AlertLogs`
- ก่อนแก้: 1 job ต่อ 4 ชม. = ~48 alerts → หลังแก้: 1 job ตลอดอายุ = 2 alerts (ลด ~96%)
- สร้าง proposal + task ที่ `.cospec/plan/changes/fix-sla-alert-dedup/`
- Implement 3 อย่าง:
  1. **Resolve merge conflict** — เลือกเวอร์ชัน stashed (มี IOptionsMonitor + configurable options)
  2. **เพิ่ม ConcurrentDictionary dedup** — key format `"CM:123:SLA-WARN"` / `"PM:456:SLA-OVER"`
  3. **เพิ่ม PM age filter** — `AND pm."CreateDate" >= NOW() - (@PmAlertMaxAgeHours * INTERVAL '1 hour')`
- ตรวจสอบหลังแก้: dedup 3 ชั้น (memory + SQL NOT EXISTS + advisory lock) + ไม่มีแหล่งอื่นยิง SLA alert
- ยืนยันผล: แต่ละ job แจ้งเตือนสูงสุด 2 ครั้งตลอดอายุงาน (WARN 1 + OVER 1), ไม่มี Noisy Alert แล้ว

## สรุปงานแก้วันนี้ (สิ่งที่แก้ / วิธีการแก้ / สิ่งที่ได้เรียนรู้)

### สิ่งที่แก้

- แก้ `SLAMonitorWorker.cs` ที่มี merge conflict ค้างอยู่ 4 จุดจนเสี่ยง compile/behavior เพี้ยน
- แก้ปัญหา `SLA alert` ยิงซ้ำงานเดิมด้วยการเพิ่ม dedup key คงที่ระดับ job+label
- แก้ query ฝั่ง PM ให้มี age filter เพื่อไม่สแกนงานเก่ามากเกินจำเป็นทุกรอบ
- แก้ configuration readiness โดยผูก `SlaAlertMonitorOptions` และเตรียมเปิดใช้ผ่าน appsettings
- แก้ mindset/analysis ของ incident เดิมจาก "น่าจะ duplicate" ให้เป็นข้อสรุปที่ถูกกว่า: flood หลักมาจาก `PM backlog burst`
- แก้ `fCalcNetSLA` ให้คำนวณ downtime เป็นระดับ seconds แทน minute-truncation
- แก้ hold rule ให้หักได้เฉพาะ `HoldType IS NULL OR HoldType = 0`; `Hold by customer` ไม่นับลด SLA
- แก้ contract ของ downtime persistence ให้ช่อง string เป็น `hh:mm:ss`, ช่อง `...Int` เป็น seconds, และ `IsOverSLA`/`SLAPercent` derive จากผลคำนวณจริง
- แก้การเลือก SQL write path โดยยืนยันว่า `fJobSlaDowntimeUpd` ที่ตรงกับ NOC caller คือ overload 16 parameters

### วิธีการแก้

- เริ่มจากไล่ flow แจ้งเตือน end-to-end (`SLAMonitorWorker` -> `spAlertInsert` -> `AlertQ/AlertTo/AlertLogs`) เพื่อหา root cause
- resolve conflict โดยเลือกโค้ดที่มี `IOptionsMonitor` และ options ครบกว่าเป็นฐาน
- เพิ่ม `ConcurrentDictionary` เป็น in-memory dedup ชั้นแรก ด้วย key แบบ `JobType:JobId:Label`
- คง SQL `NOT EXISTS` ไว้เป็น safety net ชั้นสอง และใช้ advisory lock กันยิงชนกันข้าม instance
- แยกการวิเคราะห์ให้ชัดระหว่าง `duplicate/noisy` กับ `burst จากงานจริง` ก่อนสรุปผล
- ฝั่ง alert ยึด logic หลัง hardening วันที่ `2026-04-22` เป็น current truth: config-driven worker, age filter, `MaxAlertsPerScan`, dedup 10 นาที
- ฝั่ง downtime ย้อนดูทั้ง memory + rollout เดิมก่อน เพื่อไม่สรุปปนกันระหว่าง `fCalcNetSLA` (calculator) กับ `fJobSlaDowntimeUpd` (write-back)
- ใช้ local PostgreSQL verification และ knowledge จาก `pg_proc` / `pg_get_functiondef` เพื่อตรวจ overload กับ deployed signature แทนการเดาจากชื่อไฟล์
- เช็ค caller mapping จริงใน `STS-NOC/Repositories/JobRepository.cs` ว่า path ที่ใช้งานอยู่เรียก 16 parameters ไม่ใช่ 14
- เวลาจด note แยก `historical once-per-job discussion` ออกจาก logic ปัจจุบันหลัง hardening เพื่อไม่ให้ AI ตัวถัดไปอ่านแล้วเข้าใจผิด

### สิ่งที่ได้เรียนรู้

- dedup ที่อิงข้อความ `Action` โดยตรงไม่ทน เพราะข้อความเปลี่ยนตามเวลา ต้องใช้ key ที่ stable
- dedup ที่ผูกกับสถานะ `Unread` อย่างเดียวไม่พอ เพราะผู้ใช้กดอ่านแล้ว event เดิมอาจถูกยิงใหม่ได้
- ระบบที่ดูเหมือน "ส่งเยอะ" ไม่ได้แปลว่า bug เสมอไป อาจเป็น burst จากงานจริงจำนวนมาก
- การกัน noisy ที่เสถียรควรมีหลายชั้น: memory (เร็ว), SQL (persistent), lock (concurrency)
- `AlertQ` คือ event table ส่วน `AlertTo` / `AlertLogs` เป็น fan-out/log downstream เลยโตมากกว่าได้ปกติ
- `MaxAlertsPerScan` คุม "จำนวนงานต่อรอบ" ไม่ได้คุม "จำนวนผู้รับ"
- ถ้าจะให้ downtime ตรง business จริง ต้องคิดเป็น seconds แล้วค่อย format เป็น `hh:mm:ss` ตอน persist/display
- กฎ hold สำคัญมาก เพราะถ้าหัก `Hold by customer` ผิด ระบบจะดู SLA ดีปลอมทันที
- PostgreSQL overload เป็นกับดักจริง ถ้าไม่ดู caller signature กับ `pg_proc` มีสิทธิ์แก้ผิด function
- เวลาสรุปงาน SLA ให้ AI ตัวถัดไป ต้องแยก 2 เรื่องออกจากกันเสมอ: `alert behavior/hardening` กับ `downtime SQL persistence`
